# Intake Playbooks — два режима discovery

> Пошаговый процесс Фазы −1 для двух ситуаций. Каждый шаг покрывает пункты
> чек-листа захода на клиента (5 базовых правил + 5 усилителей У1–У5).

---

# OUTSIDE-IN — холодный заход на чужой бизнес

> Когда: нет контакта или цифр клиента. Работа по публичным данным. Цель — отчёт,
> который показывает «вижу бизнес насквозь даже снаружи» + крючок «дайте цифры → точнее».

### Шаг 0 — База (анкета режим A)
Заполнить outside-in часть анкеты самостоятельно. **Минимальный якорь = город + тип
бизнеса** — этого хватает, чтобы стартовать карту конкурентов (`places_search` по
«тип + город»). URL / Google Maps / соцсети — если есть, отлично; если НЕТ — это **не
тупик, а находка** («клиента почти нет онлайн» = причина терять заявки). Чего нет
онлайн → фиксируется как находка и уходит вопросом клиенту (`CLIENT_QUESTIONS.md`).
Даже без якоря вообще (не ясны и город/тип) — не падать: собрать что есть, остальное
добрать через клиента. (Раньше тут стоял жёсткий блок «без URL+Maps нельзя» — снят:
у реального клиента онлайн-следа может не быть вовсе.)

### Шаг 1 — Фактура (НЕ результат сначала)
- `web_fetch` сайта клиента → структура, услуги, воронка записи, позиционирование.
  **Нет сайта?** Не блокер — это находка №1 (нет витрины и воронки) → в отчёт и в
  `CLIENT_QUESTIONS.md`; фактуру о клиенте добираем из Maps/соцсетей/со слов клиента.
- `places_search` по **городу + типу бизнеса** (гео-смещение + мульти-запрос) →
  конкуренты в радиусе (прямые + смежные): рейтинг, отзывы, часы, фишки. Это
  работает, даже когда у самого клиента нет ни сайта, ни Maps — рынок виден по
  «тип + город». **Клиента нет на Maps?** Тоже находка (невидим в локальном поиске).
- **Датировать** этот сбор (У4) — данные протухают.

### Шаг 2 — Углубление фактуры (усилители — не пропускать)
- **Цены конкурентов** (У1): `web_fetch` страниц конкурентов → сравнение прайса.
  Частая ошибка — пропустить этот шаг; это коммерческий рычаг, собирать всегда.
- **Дословные цитаты из отзывов** (У2): и хвалебные про клиента, и жалобы на
  конкурентов. Не пересказ — точные фразы. Идут в позиционирование.
- **Визуальный рендер** (У3): отрендерить сайт клиента и 1–2 конкурентов как
  картинки → сравнить мобильную версию и воронку записи.

### Шаг 3 — Гипотезы, помеченные как гипотезы (У5)
Сформулировать красные флаги (часы / позиционирование / SEO / воронка). Каждый —
явно «гипотеза, проверяемая цифрами», не факт. Дисциплина факт-vs-гипотеза.

### Шаг 4 — Синтез в 5 блоков
Заполнить 5 doc-шаблонов discovery:
1. DISCOVERY_PAIN — боль (из отзывов, дословно)
2. COMPETITOR_MAP — карта конкурентов + где они слабы
3. POSITIONING_ONE_LINER — одна фраза из контраста (где конкуренты слабы, а клиент силён)
4. TONE_VOICE — тон (снаружи ограничен; отметить, что глубже будет с контактом)
5. SEO_SCAFFOLD — локальные ключи, Google-профиль, что чинить

### Шаг 5 — Отчёт + крючок
Собрать отчёт (PDF консалтингового уровня). В нескольких местах — маркер «что
разблокируют реальные цифры». Финал — предложение платной персонализированной версии.
**Не брать первый драфт** (Правило №5): проверить вёрстку, где провисает.

### Gate −1→0
Позиционирование одной фразой + конкурентный разрыв названы; SEO-каркас есть;
гипотезы помечены. → можно в Phase 0.

# INSIDE-OUT — свой бренд / клиент на контакте

> Когда: есть носитель бизнеса, готовый отвечать. Цель — добыть боль/тон/цифры из
> первоисточника, чего снаружи не видно.

### Шаг 0 — База (анкета режим B)
Отправить inside-out часть анкеты клиенту. Главное — B2 (боль) и B4 (голос бренда)
записывать ДОСЛОВНО.

### Шаг 1 — Слушать язык (это ядро inside-out)
Боль и тон добываются из СЛОВ носителя, не выдумываются. Ловить характерные фразы
в ответах B2/B4 — именно они становятся позиционированием и тоном бренда.

### Шаг 2 — Фактура снаружи (дополняет, не заменяет)
Даже с контактом — собрать конкурентов (`places_search` по городу+типу) и аудит
сайта (`web_fetch`), чтобы видеть клиента в рынке. Датировать (У4). **Нет у клиента
сайта/Maps/соцсетей?** Не блокер — каждое отсутствие фиксируется как находка, а
рынок всё равно виден по «тип + город»; недостающее уточняется у клиента (он же на
контакте — спросить напрямую или через `CLIENT_QUESTIONS.md`).

### Шаг 3 — Цифры (если дал — точно; если нет — гипотезы)
Из B5: средний чек, поток, загрузка. Есть → конкретный расчёт. Нет → гипотезы,
помеченные как гипотезы (У5).

### Шаг 4 — Синтез в 5 блоков
Те же 5 doc-шаблонов, но блоки 1/3/4 (боль/позиционирование/тон) — БОГАЧЕ, чем в
outside-in, потому что есть прямой язык носителя. Это сила inside-out.

### Шаг 5 — Результат
Для своего бренда → концепт сайта + бренд-бук.
Для клиента → отчёт + предложение. Не брать первый драфт.

### Gate −1→0
Позиционирование + тон зафиксированы из реального языка; цифры или помеченные
гипотезы; конкурентный контекст есть. → Phase 0.

---

## Главное отличие двух режимов (одной строкой)

**Outside-in:** язык добывается из ОТЗЫВОВ (чужие слова о бизнесе).
**Inside-out:** язык добывается от НОСИТЕЛЯ (слова самого бизнеса).
Метод один — «боль и тон из существующего языка, не выдумывать» — меняется источник.
